## Executive summary

i18n implementation is **functionally correct** but suffers from **substantial drift**:
- 704 keys in `en.json`, 705 in `es.json` — symmetric in count, **3 keys diverge** between EN/ES due to typos.
- **507 unique `translate=` attributes** referenced across templates and built HTML.
- **198 orphan keys per language** (28% of dictionary) — defined but never referenced anywhere.
- **2 missing keys** referenced in templates but absent from `en.json`: `contactar`, `read-more`.
- The translate engine (`assets/js/translate.js`) is well-built: localStorage persistence, URL param override, prefers `fetch()` with XHR fallback, GA4 instrumented, accessibility-aware (sets `<html lang>`, updates button alt/aria-label).

Top 3 issues:
1. **P0 — 2 missing translation keys** (`contactar`, `read-more`) silently fail in EN. When a `[translate="contactar"]` element is processed by `translate.js`, the key is not in en.json → the element keeps its hardcoded fallback text. This is a SILENT BUG: not a console error, not a build error, just "the wrong text shows in EN".
2. **P1 — 3 typo'd keys diverge between EN/ES**, causing silent translation failures one direction:
   - `bulding-a-flexible` exists in EN, not ES → when ES is active, element shows untranslated EN text.
   - `building-a-flexible` exists in ES, not EN → reverse problem.
   - `significantly-enhanced` (EN) vs `significatnly-enhanced` (ES) — same.
3. **P1 — 198 orphan keys per language (28% bloat)**. ~22 KB of unused JSON downloaded per language per visit. Most are residue from archived dossiers (DebTracker, GymTracker, Norden, etc.).

## Findings by severity

### P0
- **`contactar` referenced but not defined in en.json.** Used in `content/ctas.json` line 40-41 (`secondTranslate: "contactar"` on the "Contact Me" CTA in About section). When the page is in EN: `translate.js` iterates `[translate]` elements, finds `translate="contactar"`, looks up `translations["contactar"]` in en.json → undefined → no-op. The element keeps its hardcoded text ("Contact"). Functionally works by accident.
- **`read-more` referenced but not defined in en.json.** Likely used somewhere in dossier templates. Same silent-failure behavior. Locate with `grep -rh "translate=\"read-more\"" src/ *.html`.

### P1
- **Key parity broken between EN and ES** (3 specific cases):

  | Issue | EN key | ES key | Impact |
  |---|---|---|---|
  | Typo `bulding` (EN) vs `building` (ES) | `bulding-a-flexible` | `building-a-flexible` | If template uses `translate="bulding-a-flexible"`: EN works, ES shows untranslated. If template uses `building-a-flexible`: ES works, EN shows untranslated. |
  | Typo `significatnly` (ES) | `significantly-enhanced` | `significatnly-enhanced` | Same risk profile. |
  | Comment marker keys | `__comment` | `_____comment1`, `_____comment2` | Cosmetic, not referenced. |

- **198 orphan keys per language file** (verified via Python analysis):
  - en.json has 704 keys; 506 referenced → 198 orphans.
  - es.json has 705 keys; 506 referenced → 199 orphans (the extra one is the duplicate comment marker).
  - Sample orphans: `Welcome`, `about`, `belen-recommendation`, `chinese-hsk-1`, `el-oficio-de-escritor`, `as-a-front-end-developer`.
  - Many appear to be from archived dossiers (DebTracker, GymTracker, Norden, MyBoard, Brevo BI, Research Engine, Portfolio-Ibaifernandez).
  - **Cost:** ~22 KB of dead JSON per language per page load (uncompressed; ~7 KB Brotli-compressed). Not catastrophic but it's free to fix.

- **`assets/js/translate.js:87-99` iterates `document.querySelectorAll('*')`** to translate all `translate-<attr>` attributes (e.g. `translate-aria-label`, `translate-placeholder`). On a 192 KB HTML page, this is **N²-bounded**: every element × every attribute. Measurable cost on translate (a few hundred ms on mobile). **Recommendation:** target only `[translate-]^=` elements via `querySelectorAll('[translate-html],[translate]')` and explicit attribute names.

- **No hreflang annotations** in `<head>`. SEO best practice for bilingual sites:
  ```html
  <link rel="alternate" hreflang="en" href="https://portfolio.ibaifernandez.com/?lang=en">
  <link rel="alternate" hreflang="es" href="https://portfolio.ibaifernandez.com/?lang=es">
  <link rel="alternate" hreflang="x-default" href="https://portfolio.ibaifernandez.com/">
  ```
  Without these, search engines may rank only one language version.

- **Initial language fallback is `'en'`** (`translate.js:49`). Does not respect `navigator.language` / `Accept-Language`. Spanish-speaking visitors get English on first load unless they have `?lang=es` or have previously toggled.

### P2
- **`translate-html` uses `innerHTML`** (translate.js:75). XSS risk if JSON values ever contain attacker content. Currently safe (repo owner controls JSON). See Security audit P2 — recommendation is a CI check that JSON values don't contain `<script` or similar.

- **No locale-specific formatting** for dates, numbers, currencies. The site has experience dates ("ABR 2024 - 2025") — these stay in Spanish-style ("ABR") even when EN is active. Minor.

- **JSON files at repo root** (`en.json` 73 KB, `es.json` 78 KB) clutter the root listing. Could move to `content/i18n/{en,es}.json` or `assets/i18n/`. Pure cosmetic.

- **`translate.js` does NOT load with cache-bust query param `?v=...`** on `en.json`/`es.json` per the build process. The fetch IS done with `cache: 'no-store'` (line 13) and a hand-bumped `?v=2026-03-16` query param (line 5). Bump frequency must be manual. Replace with build-injected hash.

- **`<title>` tag does NOT switch with language.** `index.html:12` is static: `Ibai Fernández — Founder-Operator. AI Product Engineer.`. When user toggles to ES, `<title>` stays in English. Meta description stays in English. OG description stays in English. **Recommendation:** add `translate="page-title"` to `<title>` and translate.js logic to update `document.title`.

- **Meta keywords are EN-only** — same issue. Less critical for SEO (Google ignores keywords), but Bing and others still index them.

- **Language toggle button** changes icon (spain flag ↔ england flag) and aria-label/alt. ✓ But does NOT visibly indicate "currently active" — it shows the target language flag. Users may misread. **Recommendation:** add visible state like `EN | ES` text with the inactive option underlined.

- **localStorage key `portfolio_language`** — fine, but `private` browsing modes block localStorage. The fetch fallback uses URL param if available, otherwise defaults to `en`. Acceptable.

### P3
- **`__comment` keys** in JSON files. Three different keys (`__comment`, `_____comment1`, `_____comment2`) signal someone tried to use JSON for documentation. If you want comments in JSON, just put them in adjacent `.md` files; JSON proper has no comments.
- **No fallback chain** — if a key is missing in es.json, it doesn't fall back to en.json. It just keeps the original DOM text. Could be a feature (deliberate omission lets EN show through) or a bug.
- **`translate.js` lacks a "missing keys" warning** in development. Adding `if (!Object.prototype.hasOwnProperty.call(translations, textKey)) console.warn('Missing key:', textKey)` in dev mode would help catch these issues early.

### Translation engine (`assets/js/translate.js`)

**Lines:** 186
**Quality:** good, modern, defensive.

```javascript
// Initial language resolution (line 48-67)
function resolveInitialLanguage() {
  // 1. URL ?lang=en|es overrides everything
  // 2. localStorage 'portfolio_language' if set to en|es
  // 3. fallback: 'en'
  // → does NOT consult navigator.language
}
```

**Loading:**
- `fetch()` with `cache: 'no-store'`, query `?v=2026-03-16`
- XHR fallback for ancient browsers (defensive)
- Error → console.error then keep current language

**Application** (line 70-100):
```javascript
function translateElements(translations) {
  // Phase 1: [translate-html] → innerHTML  (XSS surface)
  // Phase 2: [translate]      → innerText  (safe)
  // Phase 3: ALL elements × ALL attributes → setAttribute  (N² scan)
}
```

**The N² scan** in Phase 3:
```javascript
var allElements = document.querySelectorAll('*');
allElements.forEach(function(element) {
  Array.prototype.forEach.call(element.attributes, function(attribute) {
    if (attribute.name.indexOf('translate-') !== 0 || attribute.name === 'translate-html') return;
    // ...
  });
});
```

On a 192 KB HTML page with thousands of elements, this is several thousand inner iterations. Measurable cost on mobile. **Replace with targeted CSS-attribute selector:**

```javascript
// Build a list of known translate-<attr> attributes from the dictionary
// then querySelectorAll(`[translate-aria-label],[translate-placeholder],[translate-content]`)
// Single pass per attribute name.
```

✓ Allows other code to react. Nice extensibility point.

✓ Good.

### Build-time vs runtime i18n

The current approach is **purely runtime** — JS fetches en.json or es.json on each page load. Implications:
- ✓ Simple, single HTML build per page
- ✓ Easy editorial workflow (edit JSON, deploy, done)
- ✗ FOUC: initial render shows EN, then ES text replaces it after fetch resolves on slow connections
- ✗ SEO: search engines see only the EN version (default in HTML); ES content is invisible to crawlers without server-side rendering or pre-rendered ES variants
- ✗ Initial paint includes English text even for `?lang=es` (the swap happens after JS executes)

**Alternative architectures** (out of scope but worth noting):
- Build-time: generate `index.html` + `index.es.html` from the same template, serve via Netlify redirects based on `Accept-Language`. Pros: SEO, no FOUC. Cons: doubles build output count.
- Hybrid: keep runtime swap but inline the initial language's JSON into the HTML (so first paint is correct). Easier than full pre-render.

## Recommendations (prioritized)

1. **Add the 2 missing keys** `contactar` and `read-more` to `en.json` and verify `es.json` has them too. Test by visiting EN and confirming both elements show correct text.
2. **Fix the 3 typo divergences**:
   - Decide canonical spelling (`building-a-flexible`, `significantly-enhanced`).
   - Add the canonical to whichever file is missing it.
   - Delete the typo'd version from the other file.
   - Update any template references.
3. **Add a CI step** `npm run test:i18n` that:
   - Asserts key parity between en.json and es.json (modulo `__comment*` keys).
   - Detects missing keys (used in HTML but not defined). Fails the build.
   - Detects orphan keys (defined but not used). Warns.
   - Quality guard `tests/quality-guards.sh` already does many things; add this script alongside.
4. **Add hreflang annotations** in `<head>` of `index.html` and all dossier pages.
5. **Add a missing-key console warning** in `translate.js` (gated on `window.location.hostname.includes('localhost')` for dev).
6. **Purge the 198 orphan keys.** Manually review (some may be reserved); delete confirmed dead ones. Saves ~22 KB per language.
7. **Replace `querySelectorAll('*')` scan** in `translate.js:87` with targeted selectors per known attribute.
8. **Localize `<title>`, meta description, OG description.** Add `translate="page-title"` to `<title>` and matching keys. Update `translate.js` to update `document.title` and `<meta>` content on language change.
9. **Respect `navigator.language` in initial language resolution** (`translate.js:49`). Default to ES if `navigator.language.startsWith('es')` and no localStorage/URL override.
10. **Document the bilingual code-switching convention** ("Deploy & Entrega") so future translators don't "fix" it.
