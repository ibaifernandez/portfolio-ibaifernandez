AGLAYA Kanban Desk

Kanban multi-tenant para equipos que trabajan con clientes. Separa lo que es tuyo, lo que es del equipo y lo que es del cliente — y decide en cada momento quién ve qué.

● El dolor

El problema

Las herramientas Kanban del mercado te obligan a elegir entre dos mundos. O gestionas tu trabajo personal en una app. O gestionas proyectos de equipo en otra. Y cuando entran clientes, multiplica eso por las cuentas y permisos que tienes que sostener.

  1. Mezclar todo en una sola app SaaS — pierdes control sobre los datos, te atan al vendor, y los clientes ven cosas que no deberían.
  2. Tener una herramienta por tipo de trabajo — fricción permanente, cuentas duplicadas, contexto perdido entre saltos.
  3. Auto-hospedar algo open source genérico — funciona pero te quedas sin UX, sin notificaciones, sin email digest, sin la separación cliente/equipo que necesitas.

AGLAYA Kanban Desk no te obliga a elegir. Tres tipos de espacio de trabajo en una sola UI, soberanía total sobre los datos, mecánica Kanban moderna con drag-and-drop accesible.

● Arquitectura

Cómo funciona

El producto opera sobre una jerarquía estricta de cinco niveles. Cada nivel hereda las restricciones de visibilidad del nivel superior, y el aislamiento se garantiza tanto en código como en base de datos.

01 Organization
→
02 Workspace
personal · interno · externo
→
03 Board
→
04 Column
→
05 Card

Tres tipos de workspace cubren los tres mundos: personal para tu trabajo individual, interno para proyectos de equipo, externo para espacios compartidos con clientes. Los clientes ven únicamente los workspaces externos que les hayas asignado. El equipo ve todo lo interno. Tú ves todo. Cambiar un workspace a tipo externo dispara un aviso explícito porque su visibilidad cambia.

El aislamiento no descansa sobre una sola capa. Cada petición pasa por middlewares concéntricos (requireAuth → requireRole → requireWorkspaceMember) en la API Express, y la base de datos refuerza el aislamiento con Row Level Security sobre Postgres usando funciones SECURITY DEFINER. Aunque la capa de aplicación fallara, la base de datos rechaza la consulta. El cliente nunca toca tablas Supabase directamente — solo usa supabase.auth.* para autenticación; todos los datos pasan por el backend.

● Frontera honesta

Lo que NO hace

Lo que el producto no hace o todavía no ofrece, explícito:

  • No está multi-organización todavía — single-tenant intencional, Phase 5 sin fecha (ADR-020).
  • No tiene SSO ni MFA — auth por JWT clásico con refresh token, suficiente para uso interno actual.
  • No ofrece versión auto-hospedada turn-key para terceros — código MIT y repo público, pero opera como instancia única en kanban.aglaya.biz.
  • No está libre de deuda — ~35 hallazgos abiertos + 3 parciales en backlog post-Mariana Trench (mayoría medios y bajos), todos catalogados.

La frontera no es marketing defensivo: es decisión de arquitectura. El producto sirve un caso de uso concreto (AGLAYA + clientes) y crece según evidencia de uso real, no según roadmap aspiracional.

● Detalles técnicos

Por dentro

Modelo de datos y permisos

Jerarquía estricta: Organization → Workspace → Board → Column → Card. Cinco niveles, cada uno hereda restricciones del superior. Los workspaces son tipados (personal / interno / externo) y ese tipo determina la visibilidad del contenido.

Cinco roles conviven en el sistema: superadmin (auditabilidad universal), admin (gestión macro), owner del workspace, member del workspace, cliente (solo workspaces externos asignados). Cada acción tiene matriz de permisos explícita documentada en docs/PERMISSIONS.md.

Aislamiento de datos en dos capas: middlewares Express concéntricos (requireAuth → requireRole → requireWorkspaceMember) más RLS en Postgres con políticas SECURITY DEFINER. El cliente nunca toca tablas Supabase directamente — solo el endpoint supabase.auth.* para autenticación; el resto de datos pasa siempre por el backend.

Stack y decisiones técnicas

Backend Node.js + Express 4 con separación canónica app.js (configuración + rutas + middlewares) / index.js (entry point TCP) — ADR-024. JWT con refresh token + access de 15 minutos (B-02 post-audit). Helmet con CSP en producción. Rate limiting global + específico por endpoint sensible.

Frontend React 18 + Vite + TailwindCSS. @dnd-kit para drag-and-drop con KeyboardSensor (A-02 post-audit, WCAG 2.1.1). Avatar con crop integrado. Mini-kanban decorativo determinista en cards de workspace para identidad visual sin assets externos.

Email vía Resend. Admin digest + user digest diarios. node-cron del servidor reemplazado por GitHub Actions cron (mejor SLA, sin proceso vivo en el server, ADR no formal — decisión operacional documentada en INCIDENTS.md).

Base de datos Supabase: nueve tablas, las nueve con RLS activa post-audit (B-04 cerró el caso de organizations), 17 foreign keys, 37 policies verificadas en smoke test de producción, 7 índices de rendimiento en columnas de alta frecuencia (workspace_members.user_id, notifications.user_id parcial WHERE read=false, cards.board_id, columns.board_id, boards.workspace_id, users.organization_id).

Auditoría Mariana Trench

Auditoría propia en cuatro fases entre 2026-05-27 y 2026-05-28. Cuatro dimensiones: Fase A (UX, accesibilidad, performance, SEO), Fase B (seguridad, base de datos, arquitectura), Fase C (legal — RGPD, Ley 21.719 chilena, LGPD brasilera, DPAs, cookies, retention), Fase D (DevOps, observabilidad, documentación, mantenibilidad).

A UX · A11y · Perf · SEO 24 hallazgos · 5 críticos · WCAG 2.1 AA
B Seguridad · DB · Arquitectura 19 hallazgos · 2 críticos mitigados en vuelo (XSS upload, backup ausente)
C Legal multi-jurisdicción 18 hallazgos · RGPD + Ley 21.719 + LGPD · 6 DPAs archivados
D DevOps · Observabilidad · Docs 18 hallazgos · Sentry + CI workflow + R2 backup diario

Resultado: 79 hallazgos formales, 38 cerrados post-Sprint 1 (18 críticos + 15 altos + 5 medios), ~35 en backlog catalogado + 3 parciales en cierre. Cada hallazgo lleva ID, dimensión, severidad, evidencia, mitigación y commit SHA en findings.json.

Dos críticos mitigados en vuelo durante el propio audit:

  • B-CRIT-01 · CVSS 8.0: XSS explotable vía upload de SVG. Multer sin fileFilter aceptaba MIMEs arbitrarios, el proxy /uploads/* de Netlify daba mismo origen, y el JWT en localStorage era robable. Mitigación cuádruple: extension blocklist + MIME allowlist + validación de magic-bytes vía file-type@16.5.4 + cinco tests de regresión. Commit 402b0d7.
  • B-CRIT-02: Backup ausente sobre plan Supabase Free (sin daily backups, sin PITR). Un solo DROP o migración mal aplicada significaba pérdida total. Mitigación: GitHub Actions cron diario, pg_dump con PG17 client, upload a Cloudflare R2 vía API nativa (R2 rechaza cfut_ tokens en el endpoint S3-compat), retention 30 días, runbook completo en docs/runbooks/db-restore.md. Commit final 3ae6541.

Tras el cierre, los 16 críticos abiertos del audit se cerraron en cuatro batches de Sprint 1: 8 accesibilidad (labels, KeyboardSensor, role=dialog, focus-trap, skip-link, aria-labels), 5 legales (política de privacidad propia, docs/legal/, endpoints self-delete y self-export), 3 operacionales (Sentry, CI, sincronización de SECURITY.md con la realidad post-audit). Sumados a los dos críticos mitigados en vuelo: 18 críticos remediados en total.

Compliance multi-jurisdiccional

Cobertura cruzada de tres marcos de protección de datos. Decisiones documentadas y declinadas formalmente cuando aplica — auditabilidad sobre todas las decisiones.

RGPD (Europa): Art. 13/14 cubiertos por política de privacidad propia publicada en /privacidad (distinta de la corporativa de aglaya.biz). Art. 17 (supresión) cubierto por DELETE /api/auth/me. Art. 20 (portabilidad) cubierto por GET /api/auth/me/export. Art. 28 (encargados) cubierto por 6 DPAs archivados en docs/legal/dpas/: Supabase con TIA, Resend, Railway, Netlify, Cloudflare, Microsoft.

Ley 21.719 (Chile): Art. 14 ter (información al titular) cubierto por la política y los endpoints self-service. LGPD (Brasil): Art. 9 (información) y Art. 50 (gobernanza) cubiertos por política + DPO designado (privacidad@aglaya.biz).

Decisiones declinadas formalmente: revisión legal externa de €500-1500 declinada (política aprobada in-house por audit, re-evaluable si volumen UE supera 5000 únicos/año), representante UE Art. 27 no requerido obligatoriamente al volumen actual, versión trilingüe de la política pospuesta hasta tracción real.

Observabilidad y operación

Sentry inicializado server-side (@sentry/node requerido ANTES que cualquier otro require en server/index.js). Variables de entorno SENTRY_DSN, SENTRY_ENVIRONMENT y SENTRY_RELEASE configuradas en Railway. El error middleware en server/app.js captura 5xx (Sentry.captureException) y los intentos de bypass del proxy Netlify (B-03 monitor). Sentry cliente aún no añadido.

Backup workflow en GitHub Actions: cron 17 3 * * * UTC, pg_dump con PG17 client, upload a Cloudflare R2 vía API nativa (/accounts/{id}/r2/buckets/{b}/objects/{key} con bearer cfut_; R2 rechaza esos tokens en el endpoint S3-compat, por eso fue necesario migrar de rclone y aws-cli a boto3 y finalmente a la API nativa). Retention 30 días automática.

CI workflow GitHub Actions corre npm test en cada PR. Estado actual: 102 tests verde / 4 skipped (preexistentes de auth/admin/security con mocks legacy que requieren refactor mayor; skip documentado con razón). El 4 skipped es deliberado; cualquier rojo nuevo bloquea merge.

Healthcheck /api/health con verificación real de dependencias — devuelve no-200 si Supabase o secrets críticos no responden. Cobertura por test dedicado.

Frontera honesta y backlog

~35 hallazgos abiertos + 3 parciales post-Sprint 1, todos catalogados por ID, severidad y esfuerzo en docs/backlog/audit-mariana-roadmap.md. Sprint 2 cubre los altos remanentes: a11y restantes, B-03 Railway custom domain + Cloudflare WAF, refactor de tests legacy preexistentes. Sprints 3+ cubren medios y bajos.

Hallazgos parciales con cierre pendiente al 100%: A-20 (h1 añadido en LoginPage, falta en WorkspaceDashboard / AdminPage / Toolbar / ResetPasswordPage), B-03 (monitor activo, falta custom domain Railway + Cloudflare WAF/Tunnel — runbook docs/runbooks/railway-custom-domain.md listo), C-09 (template DPIA creada; formal pendiente hasta crecimiento de usuarios o exigencia de cliente externo).
